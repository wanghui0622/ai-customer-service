# 系统架构说明

> 面向**第一次接触本仓库**的读者：快速建立全局认知，不涉及具体类名与实现细节。

---

## 1️⃣ 项目简介（1 分钟理解）

**这个项目是做什么的？**  
一个可运行的 **AI 客服 / 对话式智能服务** 示例：用户发消息，系统经编排后调用大模型并返回答案，并可按需接入知识检索（RAG）、工具调用与会话记忆。

**解决什么问题？**  
把「对话 + 检索 + 工具 + 记忆 + 提示词 + 模型调用」拆成清晰边界，便于本地演示、替换实现与逐步演进到生产形态。

**核心价值是什么？**  
👉 **用多模块 + SPI 抽象，把“编排大脑”与“可替换能力件”分开**，而不是把所有逻辑堆在一个 CRUD 服务里。

**一句话本质**  
👉 **以编排层为中心、可插拔的 LangChain4j 风格 AI 客服后端骨架。**

## 2️⃣ 整体架构图（必须）

下面从 **HTTP 入口** 一直画到 **各能力模块**（与源码模块对应）。

```
┌─────────────────────────────────────────────────────────────┐
│  Controller（可执行应用层）                                    │
│  • ai-reactive-chat：聊天 API（WebFlux，默认 8081）            │
│  • ai-admin-webmvc：管理 / 评估 / Prompt / RAG 等（MVC，8080）│
└──────────────────────────┬──────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│  AI Service（编排层 —— “大脑”）                              │
│  模块：ai-service                                            │
│  职责：唯一对话管道 — 聚合 Memory / RAG / Tools / Prompt / LLM │
└──────────────────────────┬──────────────────────────────────┘
                           ↓
        ┌──────────────────┼──────────────────┐
        ↓                  ↓                  ↓
   ┌─────────┐       ┌──────────┐       ┌──────────┐
   │ ai-core │       │ai-prompt │       │  ai-rag  │
   │ LLM 调用│       │Prompt 构建│       │ 知识检索  │
   └─────────┘       └──────────┘       └──────────┘
        ↓                  ↓                  ↓
   ┌─────────┐       ┌──────────┐       ┌──────────┐
   │ai-memory│       │ ai-tools │       │ ai-eval  │
   │上下文管理│       │ 工具执行  │       │ 评估实验  │
   └─────────┘       └──────────┘       └──────────┘
```

**各层职责简述**

| 层级 | 职责 |
|------|------|
| **Controller** | 解析 HTTP、鉴权/限流（与应用配置相关）、把请求交给编排门面；聊天流式响应在 WebFlux 侧完成。 |
| **AI Service（编排层）** | **唯一业务管道**：按固定顺序加载记忆 → 路由决策是否用 RAG/工具 → 检索与工具 → 拼 Prompt → 调 LLM → 写回记忆；返回答案（及可选调试快照）。 |
| **ai-core** | 封装 **大模型调用**（LangChain4j 等），对上暴露统一 `LlmClient` 能力；可叠加熔断、重试等韧性策略。 |
| **ai-prompt** | **提示词模板与组装**：把历史、检索片段、工具结果、当前用户问题合成一条可发给模型的文本。 |
| **ai-rag** | **知识检索**：按配置选择向量存储实现，把用户问题映射为相关文本片段列表。 |
| **ai-memory** | **会话级上下文**：按 `sessionId` 读写多轮对话摘要或原文，供 Prompt 使用。 |
| **ai-tools** | **外部工具/业务接口**：在编排允许时执行命名工具，结果进入 Prompt。 |
| **ai-eval** | **评估与实验**：依赖编排能力做批量或脚本化评测，**不是每条用户消息必经路径**（多在管理端或独立入口触发）。 |

👉 **编排层（ai-service）只依赖 SPI 接口**，具体实现分布在各子模块并由 Spring 注入，因此换实现不等于改管道代码。

## 3️⃣ 核心调用流程（重点）

**一次用户输入 → 最终回答** 的完整链路如下（与 `AiChatService` 管道一致）。

1. **用户输入**  
   客户端 `POST` 聊天接口，携带 `sessionId` 与 `message`。

2. **Memory 如何参与**  
   👉 **先读**：按 `sessionId` 加载历史文本（格式化后的对话串）。  
   👉 **后写**：模型返回后，把本轮用户话与助手答复写回存储，供下一轮使用。

3. **RAG 如何参与**  
   经 **Agent 路由** 判定是否需要检索；若开启且决策为需要，则用当前 `message` 做 **检索**，得到若干知识片段；否则片段列表为空。

4. **Tool 如何参与**  
   若编排与路由允许工具，则按决策的工具名执行 **工具调用**，得到结构化/文本结果；否则为空串。

5. **Prompt 如何构建**  
   `PromptComposer` 将 **历史 + 检索片段 + 工具结果 + 当前问题** 合成一条完整 **prompt 字符串**（模板与章节顺序由 ai-prompt 侧定义）。

6. **LLM 如何调用**  
   `LlmClient` 将上述 prompt 发给模型，得到 **answer** 文本；外围可有熔断、重试、限流等保护。

7. **可选调试信息**  
   非流式接口在允许观测配置时，可一并返回路由决策、是否走 RAG/工具、片段、工具输出与完整 prompt（生产常关闭）。

👉 **流式接口**通常只对 **最终答案** 做分块输出，不携带完整编排快照（与实现一致时以代码为准）。

## 4️⃣ 模块说明（清晰简洁）

以下仅回答：**做什么 / 输入输出 / 是否调用 LLM**。

### ai-service

- **做什么**：对话 **编排入口**（加载记忆、路由、RAG、工具、拼 prompt、调模型、存记忆）。  
- **输入输出**：输入为 `sessionId` + 用户文本；输出为助手文本，或带调试轨迹的结构化结果。  
- **是否调用 LLM**：**是**（通过 `LlmClient`）；另可通过配置让 **Agent 路由** 也使用 LLM 做 JSON 决策。

### ai-core

- **做什么**：**大模型访问层**（OpenAI 兼容等），并对调用链提供韧性扩展点。  
- **输入输出**：输入为完整 prompt 字符串；输出为模型文本。  
- **是否调用 LLM**：**是**（本模块职责核心）。

### ai-prompt

- **做什么**：**提示词工程**——模板、变量、多段拼装（系统/历史/知识/工具/当前问题）。  
- **输入输出**：输入为历史、检索列表、工具结果、用户消息；输出为 **一条字符串 prompt**。  
- **是否调用 LLM**：**否**（只生成文本；若要做「自动改写 prompt」需另接模型，非本模块默认职责）。

### ai-rag

- **做什么**：**向量化检索路径**上的知识获取（向量库可插拔：内存 / H2 / PostgreSQL / Milvus 占位等）。  
- **输入输出**：输入为用户查询；输出为 **检索到的文本片段列表**。  
- **是否调用 LLM**：**否**（嵌入与排序策略若引入模型，属于实现细节，可配置）。

### ai-memory

- **做什么**：**会话记忆**——多轮上下文持久化策略（进程内、Redis 等）。  
- **输入输出**：输入为 `sessionId` 与消息对；输出为 **供 Prompt 使用的历史字符串**。  
- **是否调用 LLM**：**否**。

### ai-tools

- **做什么**：**工具执行适配**——例如查订单、查物流等（示例可为占位 JSON）。  
- **输入输出**：输入为工具名与用户消息等；输出为 **工具结果字符串**。  
- **是否调用 LLM**：**否**。

### ai-eval

- **做什么**：**质量评估与实验流水线**（依赖编排好的 `ai-service` 做批量对比）。  
- **输入输出**：输入为评测集或脚本参数；输出为 **指标或报告型结果**。  
- **是否调用 LLM**：**间接是**（通过编排调用主对话链路时会触发 LLM，本模块自身不负责直连模型）。

## 5️⃣ 系统设计理念（非常关键）

### 为什么要这样拆模块？

👉 **解耦思想**  
检索怎么存、记忆放 Redis 还是内存、模型换哪家——都不应污染「一次对话怎么走」的主流程。主流程只认 **SPI 接口**，实现放在独立模块。

👉 **编排层思想（ai-service 是大脑）**  
所有「先记忆、再决策、再检索、再工具、再拼词、再模型、再写记忆」的顺序 **只在一处定义**。这是 **AI 系统的中枢**，而不是散落在多个 Controller 里。

👉 **AI 系统 ≠ CRUD 系统**  
传统业务是 **资源 + 状态机**；本系统是 **管道 + 策略 + 可替换子系统**。CRUD 模块化按表拆；这里按 **能力边界** 拆（模型、记忆、知识、工具、提示词、评估）。

## 6️⃣ 技术选型说明

| 选型 | 说明 |
|------|------|
| **Spring Boot** | 统一配置、依赖注入、可执行 jar、生态成熟，便于接 Actuator、Security、Micrometer。 |
| **WebFlux（聊天应用）** | **响应式栈 + Netty**，适合 **SSE/流式** 输出与高并发 IO；聊天服务独立为 `ai-reactive-chat`，与阻塞式管理端分离。 |
| **LangChain4j** | Java 侧 **LLM 抽象与 OpenAI 兼容实现**，降低对接成本，便于切换模型供应商。 |
| **H2 + 内存存储** | **默认轻量**：会话记忆可用 **进程内** 实现；RAG 向量默认可 **in-memory**，教学环境也可用 **H2** 模拟持久化向量表（小数据量、检索在应用层算相似度）。 |

### 如何替换为 MySQL / Redis / Milvus？

👉 **MySQL**  
- 业务元数据、评测记录、运营配置等可用关系库；若要做 **会话历史表**，可实现新的 `MemoryStore` 或等价持久层并注入。  
- RAG 侧若用 **PostgreSQL** 存向量，项目已提供 JDBC 方向实现思路；换 MySQL 需自行评估向量能力与索引方案（通常更推荐 PG + pgvector 或专用向量库）。

👉 **Redis**  
- 配置 Spring Data Redis 后，使用 **Redis 会话记忆** 实现多实例共享会话（`application.yml` 中有预留注释）。  

👉 **Milvus**  
- 将 `aics.rag.vector-store` 配置为 `milvus`，并实现/完善 **Milvus 向量存储适配器**（仓库中可能含占位实现，生产需接官方 SDK 与索引策略）。

## 7️⃣ 项目目录结构（帮助上手）

多模块 Maven 工程（根 `pom.xml` 聚合），除下列外还有 **`ai-parent`（BOM）**、**`ai-common`**（公共工具）、**`ai-model`**（领域/DTO）、**`ai-agent-router`**（路由决策）等支撑模块。

| 目录/模块 | 作用 |
|-----------|------|
| `ai-parent` | 统一版本与插件管理 |
| `ai-common` | 通用工具与常量 |
| `ai-model` | 领域模型与传输对象 |
| `ai-core` | LLM 客户端与相关抽象 |
| `ai-prompt` | 提示词模板与构建 |
| `ai-rag` | 检索与向量存储实现 |
| `ai-memory` | 会话记忆与存储后端 |
| `ai-tools` | 工具执行扩展 |
| `ai-eval` | 评估与演示入口 |
| `ai-agent-router` | Agent 路由（规则或 LLM） |
| `ai-service` | **编排与门面（核心）** |
| `ai-reactive-chat` | **可执行**：聊天 WebFlux |
| `ai-admin-webmvc` | **可执行**：管理端 MVC |

## 8️⃣ 快速运行指南

### 启动

- **管理端（Prompt / RAG / Memory / Eval 等，默认 8080）**  
  `mvn -pl ai-admin-webmvc spring-boot:run`

- **聊天服务（默认 8081）**  
  `mvn -pl ai-reactive-chat spring-boot:run`

### 访问接口

- **聊天（JSON 全量或带轨迹）**  
  `POST http://localhost:8081/api/chat`  
  Body：`{"sessionId":"...","message":"..."}`  

- **聊天（流式）**  
  `POST http://localhost:8081/api/chat/stream`  
  `Accept: text/event-stream`  

- **健康检查（管理端示例）**  
  `GET http://localhost:8080/health`  

👉 模型 **API Key、Base URL** 等请用环境变量或本地配置覆盖，**勿提交密钥**。

### 如何测试聊天

1. 启动 `ai-reactive-chat`。  
2. 用 curl、HTTP 客户端或配套前端对 `/api/chat` 发 JSON。  
3. 观察返回文本；若开启 `expose-prompt-trace`，可在 JSON 响应中看到编排调试字段（生产建议关闭）。

## 9️⃣ 一句话总结（收尾）

👉 **本项目是一个以编排层为核心、SPI 可插拔的 AI 客服后端示例，用 Spring Boot + LangChain4j 串起记忆、检索、工具与模型调用，并通过双应用（MVC 管理 + WebFlux 聊天）分离关注点。**
